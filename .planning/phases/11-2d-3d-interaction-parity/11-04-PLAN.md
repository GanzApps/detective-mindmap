  <name>Task 2: Fix 3D drag vertical inversion — negate deltaY in world-space formula</name>
  <files>src/components/graph/MindMap3D.tsx</files>
  <action>
Open `src/components/graph/MindMap3D.tsx`.

Find the drag delta block inside the `onMouseMove` handler (around lines 503-510):

```ts
const yaw = cameraRef.current.rotY;
const zoomFactor = Math.max(cameraRef.current.zoom, 0.35);
const worldDeltaX = ((deltaX * Math.cos(yaw)) - (deltaY * Math.sin(yaw))) / zoomFactor;
const worldDeltaY = ((deltaX * Math.sin(yaw)) + (deltaY * Math.cos(yaw))) / zoomFactor;
```

The bug: `projection3d.ts` computes screen Y as:
`sy = viewport.height / 2 + offsetY - (ty * scale)`
World Y is negated when projected to screen. Therefore dragging the mouse down
(+deltaY in screen space) should decrease world Y, not increase it. The current
`worldDeltaY` formula adds `+deltaY*cos(yaw)` which goes in the wrong direction.

Replace ONLY the two `worldDeltaX` / `worldDeltaY` lines with:

```ts
const worldDeltaX = ((deltaX * Math.cos(yaw)) + (deltaY * Math.sin(yaw))) / zoomFactor;
const worldDeltaY = ((deltaX * Math.sin(yaw)) - (deltaY * Math.cos(yaw))) / zoomFactor;
```

Key change: in `worldDeltaX` the sign on the `deltaY * sin(yaw)` term flips from
minus to plus; in `worldDeltaY` the sign on the `deltaY * cos(yaw)` term flips
from plus to minus. This compensates for the screen-Y inversion so that dragging
a node up moves it up, and dragging it down moves it down.

Do NOT change the pan or rotation branches (`dragModeRef === 'pan'` /
`dragModeRef === 'rotate'`). Those operate directly in screen/camera space and are
not affected by the projection inversion.

  <name>Task 3: Fix 3D→2D position carry — convert world (x, z) to canvas (x, y) before sharing</name>
  <files>src/components/graph/GraphWorkspace.tsx</files>
  <action>
Open `src/components/graph/GraphWorkspace.tsx`.

Background: `SharedNodePosition` has fields `{ x, y }`. In 3D, these map to
world-X and world-Z respectively (see `projection3d.ts` lines 119 and 122:
`x: sharedPosition.x`, `z: sharedPosition.y`). In 2D, `ForceGraph2D.tsx` applies
`override.x` and `override.y` directly as 2D canvas coordinates (force simulation
world space). The two coordinate spaces are unrelated — 3D world units and 2D force
simulation units are scaled differently — so a position from 3D drag appears at
the wrong location (or far off-canvas) in 2D.

The fix is a coordinate conversion applied only when the position update originates
from the 3D renderer. The simplest correct mapping is:
- 2D canvas X  ←  3D world X  (same axis, but 3D uses radii ~150-400 units while
  2D force layout uses similar pixel-scale units, so a scale factor is needed)
- 2D canvas Y  ←  3D world Z  (3D world Z maps to lateral spread in the 2D plane)

The 3D layout uses `getTierRadius` values ranging from 0 (root) to ~400 for outer
tiers. The 2D force layout distributes nodes across the canvas (typically 600-900px
wide/tall). A scale factor of `1.5` applied to both axes brings them into
compatible visual ranges.

Steps:

1. Locate the existing `handleUpdateNodePosition` callback (around line 188):

```ts
const handleUpdateNodePosition = useCallback((nodeId: string, position: SharedNodePosition) => {
  setNodePositions((current) => {
    ...
    return { ...current, [nodeId]: position };
  });
}, []);
```

2. Add a new conversion helper directly above that callback:

```ts
// Converts a position that originated from the 3D renderer into 2D canvas
// coordinate space. In 3D, SharedNodePosition.x = world X and
// SharedNodePosition.y = world Z. Both axes use the same scale in 2D.
// A factor of 1.5 maps the 3D tier-radius range (~0-400) to the 2D force
// layout canvas range without clamping.
function convert3DPositionTo2D(position: SharedNodePosition): SharedNodePosition {
  const SCALE = 1.5;
  return { x: position.x * SCALE, y: position.y * SCALE };
}
```

Place this as a module-level function (outside the component), right above the
`GraphWorkspace` component definition.

3. Add a `viewMode` parameter to the `handleUpdateNodePosition` callback so it can
apply the conversion conditionally. Because `viewMode` is already in scope in the
component, capture it via the `useCallback` deps array:

```ts
const handleUpdateNodePosition = useCallback((nodeId: string, position: SharedNodePosition) => {
  const stored = viewMode === '3d' ? convert3DPositionTo2D(position) : position;
  setNodePositions((current) => {
    const existing = current[nodeId];
    if (existing && existing.x === stored.x && existing.y === stored.y) {
      return current;
    }
    return { ...current, [nodeId]: stored };
  });
}, [viewMode]);
```

4. `viewMode` comes from `useCaseStore` and is already in scope. Add it to the
`useCallback` deps array as shown above.

Do NOT change the prop name `onUpdateNodePosition` in the component interfaces —
only the implementation of `handleUpdateNodePosition` in `GraphWorkspace.tsx`
changes.

Do NOT change `ForceGraph2D.tsx` or `MindMap3D.tsx` — the conversion belongs at
the shared position store boundary in `GraphWorkspace`.
